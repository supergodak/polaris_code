# Feature Specification: AI-Driven Coding Agent CLI

**Feature Branch**: `001-ai-coding-agent-core`
**Created**: 2026-03-20
**Status**: Draft
**Input**: ローカルLLM（MLX）で動作する完全AI駆動のコーディングエージェントCLIツール

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 自然言語でコーディングタスクを指示する (Priority: P1)

ユーザーはターミナルでエージェントを起動し、「認証機能を追加して」のように自然言語で指示を出す。エージェントはコードベースを自律的に探索し、関連ファイルを読み、実装計画を立て、コードを生成・編集し、結果を報告する。人間はdiffを確認して承認するだけ。

**Why this priority**: これがプロダクトの根幹。この体験が成立しなければ他の機能は意味がない。

**Independent Test**: エージェントを起動し、簡単なタスク（例: 「hello worldスクリプトを作成して」）を指示し、ファイルが正しく生成されることを確認。

**Acceptance Scenarios**:

1. **Given** エージェントが起動している, **When** ユーザーが「src/hello.pyにhello world関数を作成して」と入力, **Then** エージェントがファイルを作成し、内容を表示して承認を求める
2. **Given** エージェントが起動している, **When** ユーザーが「テストを実行して」と入力, **Then** エージェントがpytestを実行し、結果を表示する
3. **Given** エージェントが起動している, **When** ユーザーが曖昧な指示を出す, **Then** エージェントが明確化のための質問を返す

---

### User Story 2 - エージェントが自律的にコードベースを探索する (Priority: P1)

エージェントはタスク遂行のために、ファイル検索（glob）、テキスト検索（grep）、ファイル読み取りを自律的に行い、コードベースの構造と内容を理解する。人間が「このファイルを見て」と指示する必要はない。

**Why this priority**: 自律的な探索能力がなければ、人間がファイルパスを逐一教える必要があり、AI駆動の価値がない。

**Independent Test**: 既存のPythonプロジェクトでエージェントを起動し、「このプロジェクトの構造を説明して」と指示し、正確な構造説明が返ることを確認。

**Acceptance Scenarios**:

1. **Given** Pythonプロジェクト内でエージェントが起動, **When** 「main関数はどこにある？」と質問, **Then** エージェントがgrepでmainを検索し、該当ファイルと行番号を回答
2. **Given** エージェントが起動, **When** 「テストファイルを全て列挙して」と指示, **Then** エージェントがglobを実行し結果を表示

---

### User Story 3 - 複数ターンの対話でタスクを進める (Priority: P2)

ユーザーとエージェントが対話形式でタスクを進める。エージェントの提案に対してフィードバックを返し、修正を重ねて目的のコードに到達する。

**Why this priority**: 一発で完璧なコードが生成されることは稀。対話による反復改善がAI駆動開発の本質。

**Independent Test**: エージェントにコード生成を依頼し、「メソッド名をsnake_caseに変えて」のようなフィードバックを与え、正しく反映されることを確認。

**Acceptance Scenarios**:

1. **Given** エージェントが関数を生成済み, **When** ユーザーが「エラーハンドリングを追加して」と指示, **Then** エージェントが既存の生成コードを修正し、差分を表示
2. **Given** 会話履歴がある, **When** ユーザーが「さっきの変更を元に戻して」と指示, **Then** エージェントが以前のバージョンに戻す操作を提案

---

### User Story 4 - LLMバックエンドを切り替える (Priority: P3)

ユーザーは設定やコマンドラインオプションで使用するLLMモデルを切り替えられる。MLXのローカルモデル間（Qwen2.5-Coder-32B ↔ CodeLlama-70B等）の切り替えが可能。

**Why this priority**: 初期はMLXの単一モデルで十分だが、比較検証やモデル進化に対応するため差し替え可能な設計は重要。

**Independent Test**: 設定ファイルでモデル名を変更し、エージェント再起動後に別のモデルで動作することを確認。

**Acceptance Scenarios**:

1. **Given** 設定ファイルにmodel指定がある, **When** エージェントを起動, **Then** 指定モデルがロードされて使用される
2. **Given** 別のモデル名に設定を変更, **When** エージェントを再起動, **Then** 新しいモデルで動作する

---

### Edge Cases

- LLMがツール呼び出しに失敗した場合（不正なJSON出力等）→ リトライ、最大3回で打ち切り
- ツール実行がタイムアウトした場合 → タイムアウトを報告しLLMに次の判断を委ねる
- コンテキストウィンドウを超える大きなファイル → 先頭N行のみ読み取り、残りの存在を通知
- エージェントが無限ループに陥った場合 → 最大反復回数（デフォルト25）で強制停止
- モデルのロードに失敗した場合 → エラーメッセージと対処法を表示
- 破壊的操作（rm -rf等）をLLMが提案した場合 → 必ずユーザー確認、警告表示

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムはターミナル上でインタラクティブなチャットインターフェースを提供しなければならない
- **FR-002**: システムはMLXを通じてローカルLLMモデルをロードし推論を実行できなければならない
- **FR-003**: システムはReActパターン（思考→ツール実行→観察→次の行動）でエージェントループを実行しなければならない
- **FR-004**: システムは以下のツールを提供しなければならない: read_file, write_file, edit_file, glob, grep, bash, ask_user
- **FR-005**: ファイル書き込み・編集操作はユーザー承認後に実行されなければならない
- **FR-006**: bash実行はユーザー承認後に実行されなければならない
- **FR-007**: システムは会話履歴を保持し、文脈を踏まえた応答を生成しなければならない
- **FR-008**: システムは設定ファイルによるモデル・パラメータの変更に対応しなければならない
- **FR-009**: LLMのツール呼び出し出力が不正な場合、リトライまたはエラー回復を試みなければならない
- **FR-010**: エージェントループの最大反復回数を設定可能とし、無限ループを防止しなければならない

### Key Entities

- **Conversation**: ユーザーとエージェント間のメッセージ履歴。ユーザーメッセージ、アシスタントメッセージ、ツール呼び出し・結果を含む
- **Tool**: エージェントが使用できる操作の定義。名前、説明、パラメータスキーマ、実行関数を持つ
- **LLMBackend**: モデルのロード、推論実行、ツール呼び出しパースを担当する抽象レイヤー
- **AgentLoop**: ReActパターンを管理するコアエンジン。思考・行動・観察のサイクルを制御
- **Permission**: 操作の承認状態を管理。自動許可、都度確認、拒否の3レベル

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーが指示を出してから、エージェントが最初のツール実行を開始するまで30秒以内
- **SC-002**: 「ファイル作成」「ファイル編集」「テスト実行」の基本タスクの自律完了率80%以上
- **SC-003**: ツール呼び出しの不正出力に対するリカバリー成功率70%以上
- **SC-004**: 32Bモデル使用時のメモリ使用量64GB以下（128GB環境の半分以下）
- **SC-005**: エージェントの起動（モデルロード含む）が60秒以内

## Assumptions

- ユーザーはApple Silicon Mac（M5 Max 128GB）を使用する
- MLXフレームワークがインストール済みで、対応モデルがダウンロード済み
- ユーザーはターミナル操作に習熟している
- 対象プロジェクトはローカルファイルシステム上にある
- インターネット接続は不要（全てローカルで完結）

## Non-Goals

- WebブラウザやGUIベースのインターフェース
- リモートAPIへの接続（初期フェーズ）
- マルチユーザー対応
- IDE/エディタプラグイン
- コード補完（インライン補完）機能
- 人間がコードを直接編集するワークフローの支援
